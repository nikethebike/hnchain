# HNChain Constitution

Status: Draft

Version: 0.1.0

Date: 2026-07-18

## Purpose

The HNChain Constitution is not a legal document.

It is a set of protocol principles that guide long-term governance,
specification design, implementation review, and ecosystem coordination.

Changing this document should be harder than changing ordinary specifications.

## Article I: Open Protocol

HNChain is an open protocol.

No company, foundation, client team, wallet, explorer, infrastructure provider,
or validator group owns the protocol.

## Article II: Security First

Security takes priority over speed, convenience, and marketing claims.

Protocol changes that weaken safety, determinism, or auditability require
extraordinary justification.

## Article III: Specification First

Protocol behavior is defined by accepted specifications.

Implementation code must not become the accidental source of protocol truth.

## Article IV: Backward Compatibility

HNChain should preserve backward compatibility whenever possible.

Backward-incompatible changes require explicit rationale, migration plans,
support windows, and activation rules.

## Article V: Transparency

Protocol changes must be discussed through documented, reproducible, public
processes.

Security-sensitive embargoes may exist, but they must have clear scope and
post-resolution disclosure rules.

## Article VI: Minimal Core

The protocol core should remain minimal.

New features must justify why they belong in the protocol instead of the
application or ecosystem layer.

## Article VII: Independent Implementations

HNChain should support independent client implementations.

Compatibility is measured by specifications and tests, not by cloning one
reference client.
